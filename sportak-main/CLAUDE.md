## Project overview

**Firemní sportovní výzvy** — internal web app for company sports challenges.
Czech UI. Users log in via Slack OAuth, record activities, and compete on a leaderboard.

See [PRD_Sportovni_vyzvy.md](PRD_Sportovni_vyzvy.md) for full product requirements.
See [design-mockup.html](design-mockup.html) and [sportovni_vyzvy_layout.html](sportovni_vyzvy_layout.html) for visual references.
The files `index.html`, `app.js`, `styles.css` are the **original prototype** — keep them as reference, do not delete.

## Running locally

```bash
# 1. Install dependencies
npm install

# 2. Set up environment variables
cp .env.local.example .env.local
# Fill in SLACK_CLIENT_ID, SLACK_CLIENT_SECRET, NEXTAUTH_SECRET, etc.

# 3. Create the local SQLite database and apply migrations
npm run db:generate
npm run db:migrate

# 4. Start the dev server
npm run dev
# Open http://localhost:3000
```

> Slack OAuth requires HTTPS for the redirect URL even in local dev.
> Use [ngrok](https://ngrok.com) or similar: `ngrok http 3000`
> Then set `NEXTAUTH_URL=https://<ngrok-subdomain>.ngrok.io` in `.env.local`

## Database schema

Defined in [db/schema.ts](db/schema.ts). Tables:

| Table | Purpose |
|---|---|
| `users` | All users. `role`: `participant` \| `admin` |
| `challenges` | Challenges. `status`: `DRAFT` \| `ACTIVE` \| `CLOSED` \| `ARCHIVED` |
| `activity_catalog` | Admin-managed list of scoreable activities |
| `enrollments` | Which users joined which challenge |
| `activities` | Logged activity entries with pre-calculated points |
| `audit_log` | Immutable log of admin interventions on other users' data |

### DB commands

```bash
npm run db:generate   # generate migration files from schema changes
npm run db:migrate    # apply pending migrations
npm run db:studio     # open Drizzle Studio (local DB browser)
```

## Auth and roles

- Login: Slack OAuth only (`/login` → `signIn("slack")`)
- Two roles: `participant` (default) and `admin`
- **Role is always read from DB** — never trusted from the session JWT alone
- First admin(s): set `ADMIN_SLACK_IDS=U123,U456` in env. Role is assigned on first login only. After that, admins manage roles via the admin panel.
- Single workspace: set `SLACK_TEAM_ID` to restrict logins to one Slack workspace

### API route pattern

```typescript
import { requireAuth, requireAdmin } from "@/lib/permissions"

export default async function handler(req, res) {
  const user = await requireAuth({ req, res })   // or requireAdmin
  if (!user) return                              // response already sent
  // ... handler logic
}
```

## Access control rules

| Action | Participant | Admin |
|---|---|---|
| View ACTIVE / CLOSED / ARCHIVED challenges | Yes | Yes |
| View DRAFT challenges | No | Yes |
| Join active challenge | Yes | Yes |
| Log / edit / delete **own** activities | Yes | Yes — no audit log |
| Edit / delete **others'** activities | No | Active challenge only + audit log |
| Manage challenges & catalog | No | Yes (not CLOSED / ARCHIVED) |
| View audit log | Yes | Yes |

### Audit log rule

```typescript
// lib/permissions.ts
isAdminIntervention(actorId, targetUserId)
// true  → write audit log entry
// false → self-edit, no audit log needed
```

## Key business rules

- Activities can only be logged within the challenge's `startDate`–`endDate` range (enforced server-side, not just UI)
- `points` are calculated and stored at insert time — use `calculatePoints(catalogItem, value)`
- CLOSED and ARCHIVED challenges are fully immutable — no writes, even for admins
- Audit log entries are never updated or deleted
